# APKSHIELD — Evaluation Report (Topic 1)

Generated by `analysis/build_report.py` with live numbers. GenAI-based automated analysis & 0–100 risk scoring of fraudulent banking APKs.

## 1. ML risk model — honest evaluation

Model selected by out-of-fold PR-AUC: **lightgbm**. Trained on the Drebin-215 dataset (5,560 malware / 9,476 benign, 215 static features).

| Model | ROC-AUC | PR-AUC |
|---|---|---|
| random_forest | 0.9964 | 0.9951 |
| xgboost | 0.9977 | 0.9965 |
| lightgbm ⭐ | 0.9978 | 0.9967 |

**Operating point (max-F1 on out-of-fold predictions):**

- Confusion matrix: TP=5382, FP=83, FN=178, TN=9393
- Recall (TPR): **96.8%** · False-positive rate: **0.9%**
- Precision on the (unrealistic) 37.0% -malware dataset: 98.5%
- **Precision prior-shifted to a realistic 10.0% malware prevalence: 92.5%** (the honest headline -- see section 3)

Top features by importance: `WRITE_EXTERNAL_STORAGE`, `android.content.pm.PackageInfo`, `Ljava.lang.Object.getClass`, `READ_PHONE_STATE`, `INTERNET`, `ACCESS_NETWORK_STATE`, `TelephonyManager.getDeviceId`, `onBind`.

## 2b. Real-world validation — live MalwareBazaar samples

4 genuine in-the-wild Android banking-trojan samples were pulled from MalwareBazaar and analyzed **in-memory** (decrypted from their password-protected archives; the malware bytes never touched disk and were never executed).

| SHA-256 (short) | Family | Static-only | Final | VT detections | GenAI |
|---|---|---|---|---|---|
| `6ca691c97ab002…` | Hook | 3 | **85 CRITICAL** | 23/75 | suspicious |
| `2b332adf4e5216…` | Hook | 14 | **85 CRITICAL** | 18/76 | malicious |
| `e29def53e832ca…` | Hook | 18 | **85 CRITICAL** | 23/76 | malicious |
| `4f181b3a42657a…` | Hook | 16 | **85 CRITICAL** | 27/76 | malicious |

**The honest finding this exposes:** the Drebin-era static ML model scores these modern trojans near zero — their malicious payload is downloaded/decrypted at runtime, so it is invisible to static features (textbook concept drift). The **threat-intel layer escalates them to CRITICAL** on multi-vendor AV consensus, and Claude independently reads them as malicious. No single layer is trusted alone: a stale static model is never allowed to clear a file that dozens of AV engines flag. This is the fraud-ops-correct behavior and the strongest argument for the layered design over a pure-ML classifier.

## 3. Evaluation honesty (what most teams skip)

- The dataset has 7779 duplicate 215-d vectors (52%) from repacked/near-duplicate apps. We cross-validate with StratifiedGroupKFold so duplicates never straddle the split — these are honest dedup-safe numbers, not the inflated random-split figure.
- No time-aware (TESSERACT) split possible: the public CSV has no timestamps. Even dedup-safe numbers overstate robustness to concept drift; treat them as upper bounds.
- Live extraction approximates the original (unpublished) Drebin feature extractor; cross-checked on known samples but not bit-identical.
- Dataset era is older than current banking trojans; the impersonation + red-flag layers exist precisely because the ML model alone cannot carry current-threat coverage.
- Precision is also reported prior-shifted to 10% malware prevalence; the raw dataset is 37% malware which inflates naive precision.

## 4. GenAI layer

- Detection stays in the fast tree + rule layers; the LLM (Claude, `claude-opus-4-8` by default) only **interprets**: plain-language verdict, MITRE ATT&CK mapping, IOC extraction, recommendations, and a Hindi customer warning.
- The model is shown **neutral evidence only** — it is never told the sample is malware and never sees our risk score, so it assesses independently (and its assessment cross-checks our score).
- **Grounding check:** every IOC the model reports must trace to an extracted fact; invented URLs/IPs/packages/hashes are caught and stripped, and surfaced in the UI as a caught hallucination.
- **Always-on fallback:** a deterministic template report runs when no API key/network is available, so the demo never breaks.

## 5. Threat intel — a scoring layer, not just a panel

VirusTotal + MalwareBazaar reputation lookups by SHA-256 (never the APK bytes), cached locally, behind API-key config flags. When several AV vendors or a MalwareBazaar catalogue entry flag a file, threat intel **escalates** the risk score (it can only raise, never lower — absence of a detection is not evidence of safety). This is what catches modern obfuscated trojans the static model misses (see section 2b).
