--- TOP TECHNIQUES ---
 - Static feature extraction with Androguard (THE baseline everyone uses): parse AndroidManifest for permissions, intents, components, signing certificate; read DEX for restricted/suspicious API calls. Build a binary feature vector (Drebin 8-category scheme; the reduced 215-feature variant is the community standard). Cheap, fast, explainable.
 - Gradient-boosted / bagged trees on tabular static features (RandomForest, XGBoost, LightGBM) - the practical sweet spot: 90-99% on standard splits, fast training, native feature importances. Use this as your primary model; the LinkedIn-described winning stack used Scikit-Learn + XGBoost. Reach for GNN/CNN/BERT only as a stretch.
 - Multi-signal COUNTERFEIT detection (the fake-BANK differentiator): don't just classify malware, PROVE impersonation. Combine (1) typosquat name similarity (Levenshtein/TheFuzz), (2) package-id keyword heuristics, (3) developer/cert-fingerprint allowlist of official bank packages, (4) icon/logo perceptual hashing (Pillow + ImageHash), (5) APK SHA-256 vs official binary. Weighted risk score 0-100.
 - Time-aware / TESSERACT evaluation (the credibility multiplier): sort by timestamp, train on past / test on future, report AUT (Area Under Time) of F1 instead of one inflated random-split accuracy. Demonstrating you avoided spatial + temporal bias beats a higher leaderboard number with expert judges.
 - Packer/obfuscation fingerprinting with APKiD as a feature AND a routing signal: packed apps make permission/API features look benign, so detect packing and (a) use it as a strong malware-correlated feature and (b) trigger a dynamic-analysis fallback.
 - GenAI context-engineering (LAMD pattern) - the standout LLM technique: collect suspicious APIs (FlowDroid/PScout/SuSi), do backward slicing to isolate only relevant instructions, then tier-wise summarize (function -> API-intent-in-context -> APK verdict). Lets a cheap GPT-4o-mini analyze whole APKs at ~$0.20/APK and hit ~90% F1, beating Drebin's 81% with far lower false-negative rate.
 - LLM hallucination grounding (mandatory reliability add-on): reconstruct variable/data dependencies and compute a Data Relationship Coverage metric against a threshold; reject/redo summaries that fail. Hallucination on obfuscated code is the dominant LLM failure mode - show a caught hallucination in the demo.
 - GenAI triage + MITRE ATT&CK mapping + risk report: feed MobSF/VirusTotal behavior to an LLM (or malwoverview --llm) to map observed TTPs to ATT&CK IDs, assign family/risk, extract IOCs, and emit an analyst report (optionally multi-agent via CrewAI). High 'so-what' value for judges.
 - Bytecode-to-image + CNN: convert classes.dex bytes to a grayscale image and classify with a lightweight CNN/ViT. Robust to API-name obfuscation and visually demoable (show the malware fingerprint + attention map), reaching 94-99.5% on standard splits.
 - Class-imbalance handling done CORRECTLY: SMOTE/SMOTEENN only inside training folds (use sklearn Pipeline). Applying resampling/scaling/feature-selection before the split is a classic leakage mistake judges look for.

--- JUDGING CRITERIA ---
 - Working end-to-end demo over benchmark numbers: judges reward an upload-APK -> static analysis -> risk score -> explainable verdict pipeline that runs live, far more than a notebook reporting 99% accuracy.
 - Domain relevance / 'so-what': clear tie to the real Indian banking-fraud problem (fake SBI/ICICI/HDFC/UPI/loan APKs spread via WhatsApp/SMS, money mules) and a deployment path (CERT-In/police reporting, takedown evidence). Pure ML without a problem narrative loses.
 - Explainability: SHAP/feature-importance or LLM-generated reasons for WHY an app is flagged. Demoing an explanation impresses more than a raw score.
 - Honest, leakage-free evaluation: precision/recall/F1/FPR + confusion matrix under a realistic ~90% benign ratio, transforms fit on train folds only, near-duplicate dedup, and ideally a time-aware (TESSERACT/AUT) split. Expert judges actively discount inflated random-split accuracy.
 - Robustness awareness: acknowledge obfuscation/packing (APKiD + dynamic fallback) and adversarial feature-perturbation evasion of linear/permission models; mention graph/hybrid features or adversarial retraining if claiming production-grade.
 - Production-grade signals: Play Integrity API, official-package + certificate-fingerprint allowlisting, MobSF/VirusTotal threat-intel integration - these read as 'deployable', not 'toy'.
 - For GenAI tracks specifically: a grounding/hallucination-check step, cost/latency transparency (e.g., ~$0.20/APK; local-LLM option for sensitive samples), and human-in-the-loop framing (force-multiplier, not full autonomy). Do NOT tell the model the sample is malicious - it biases every function as suspicious.
 - Reproducibility: runnable extraction code (Androguard), pinned dataset version + split seed, not hard-coded notebook results.

### ANGLE: Technical pipeline and tooling for automated Android APK malware analysis (static + dynamic), with concrete features, open-source repos, datasets, and ML-model 
  TECHNIQUES:
    - End-to-end APK pipeline (decompile -> feature extraction -> vectorize -> classify): Standard flow: (1) unpack APK with apktool/Androguard, decode AndroidManifest.xml from binary XML, disassemble classes.dex to smali; (2) extract static features (permissions, intents, components, API calls, opcodes, certs, URLs); (3) optionally run dynamic analysis in an emulator/sandbox to capture syscalls + network; (4) one-hot/frequency vectorize features; (5) train a classifier (RF/XGBoost/Lig
    - Drebin-style static feature set (8 categories, sparse Boolean vector): Drebin extracts 8 feature subsets: (S1) hardware components, (S2) requested permissions, (S3) app components (activities/services/receivers/providers), (S4) filtered intents, (S5) restricted API calls, (S6) used permissions, (S7) suspicious API calls, (S8) network addresses/URLs. These become a high-dimensional (~545k) Boolean presence vector; a popular reduced version uses 215 features. Linear SV
    - Opcode sequence / n-gram and image-based features: Extract the Dalvik opcode sequence from smali (via Androguard or baksmali), then use opcode n-grams or convert opcodes/bytecode to grayscale images for CNNs (e.g., BlockDroid/BIR-CNN report 99%+). Good for catching obfuscated samples where API names are hidden but instruction patterns remain.
    - API call graph + taint (source->sink) analysis with FlowDroid: FlowDroid (Soot-based, IFDS, context/flow/field/object-sensitive) builds a precise callgraph that models the Android lifecycle and tracks data flows from sources (e.g., getDeviceId, location) to sinks (e.g., sendTextMessage, HTTP). SourcesAndSinks.txt is the configurable definition file. Leaks like IMEI->SMS/network are strong malware signals.
    - Dynamic behavioral features (syscalls, binder, network): Run APK in a rooted emulator/AVD; capture: system calls via strace on zygote/app PID, binder (IPC) calls, composite high-level behaviors (CopperDroid/Droidmon), network traffic via tcpdump->pcap (analyze with CICFlowMeter for ~80 flow features or Wireshark), and logcat. Frida hooks specific Java/native APIs at runtime to log args/returns and defeat string obfuscation. Feature studies show syscalls
    - Packer/obfuscation fingerprinting as a feature: APKiD (rednaga, YARA-based 'PEiD for Android') detects compilers, packers, protectors and obfuscators from DEX/APK/ELF. Packing/obfuscation presence is itself a useful malware-correlated feature and tells the pipeline when static analysis will be unreliable and dynamic analysis is required.
    - Certificate, manifest and metadata signals: Extract signing certificate (issuer/validity/self-signed flag), package name, requested vs. declared permissions delta, exported components, minSdk/targetSdk, and embedded URLs/IPs. Self-signed certs reused across many apps and over-privileged manifests are classic heuristics.
    - Label sourcing with VirusTotal + AVClass: For building a labeled dataset, submit hashes to VirusTotal (multi-engine), then normalize the noisy vendor labels into a single family with AVClass. AndroPyTool automates this 'pre-static' labeling step before feature extraction.

### ANGLE: GenAI/LLMs for malware analysis and reverse engineering. The field splits into five concrete, build-in-a-weekend patterns: (1) LLM-assisted DECOMPILATION/REFINE
  TECHNIQUES:
    - Decompile-then-reason pipeline: LLMs cannot read binary directly. Standard flow: objdump/Ghidra/IDA for native, JADX or apktool for Android (apktool->smali, JADX->pseudo-Java). Feed decompiled/disassembled text (not raw bytes) to the LLM. LLM4Decompile-End decompiles assembly directly; LLM4Decompile-Ref/V2 REFINES Ghidra's pseudo-C (the V2 refine approach beat the end-to-end V1.5 by ~40%).
    - Context reduction via backward slicing + key-API extraction: LAMD's core trick: don't feed the whole decompiled app. Collect suspicious APIs (sensitive-data access + exfil) using PScout/SuSi/FlowDroid databases, then run a custom backward slicing algorithm (variable retrieval -> slice extraction) to isolate only instructions affecting those API calls. This solves transformer 'over-squashing' and lets a cheap GPT-4o-mini analyze whole APKs. ~$0.199/APK.
    - Tier-wise hierarchical summarization: LAMD's 3 tiers: Tier1 summarize individual sliced function/CFG behavior; Tier2 reason about API intent across calling contexts (same API benign vs malicious depending on caller); Tier3 synthesize an APK-level malicious/benign verdict with IOCs and behavioral evidence. Map-reduce over functions keeps each LLM call within context.
    - Hallucination grounding / factual-consistency check: Before trusting a function summary, have the model extract 5 data-dependency types (direct var->API, transitive call-chain, conditional control-flow, parallel contributions, derived computations) and compute a 'Data Relationship Coverage' metric against a reliability threshold. Reject/redo summaries that don't reconstruct variable dependencies. This is the single most important reliability add-on.
    - Agentic RE via MCP tool-calling: Expose disassembler internals as tools (decompile, list_functions, rename, set_comment, get_xrefs, get_callgraph) over Model Context Protocol. The LLM runs a ReAct loop: gather decompilation/xrefs, reason, then call rename/comment.     - Structured prompts with explicit tool discipline: Talos's winning prompt patterns: detailed multi-step instructions, naming conventions ('prefix renamed functions with VIBE_'), and hard rules like 'NEVER convert number bases yourself, use the convert_number MCP tool'. Keep prompts small/few-step for local models to avoid context truncation forgetting mid-task.
    - Deobfuscation -> structured JSON intel: For PowerShell/JS droppers, prompt GPT-4 to deobfuscate layered scripts and return JSON with extracted C2 servers, download URLs, and behavior intent.     - Never label the sample 'malware' in the prompt. Talos found this biases the model into flagging every function as suspicious. Ask for neutral behavior description, then judge maliciousness in a separate step.
    - Context window is the binding constraint, not model quality. Whole decompiled apps/binaries overflow context. Use slicing, key-API extraction, and map-reduce summarization. A naive 'paste the whole file' demo will truncate and 'forget' mid-
    - Resist RAG-by-default. Evidence shows generic RAG can degrade malware explanation. If you use retrieval, use a strong embedder and tight thresholds, and benchmark against a no-RAG baseline so you can justify it.

    - Permission-based ML feature vectors: Build a binary feature vector over the ~330 standard Android permissions plus counts (total permission count and 'unidentified'/proprietary permission count are consistently the top-weighted features).     - Random Forest / Gradient Boosting / XGBoost ensemble: Tree ensembles dominate because they are fast, work on tabular permission/API features, give native feature importances, and hit 90-99% accuracy on Drebin/CICMalDroid. Random Forest reported 93.1% (DroidDetective), 98-99% in SHAP studies. Winning hackathon stack (Aditya Raj) used Scikit-Learn + XGBoost.     - Multi-signal counterfeit detection (beyond ML): Strongest differentiator for fake-BANK detection specifically: don't just classify malware, prove the app is impersonating a real bank.     - Lead with a working demo, not a model metric. Upload a real fake-bank APK live and show the verdict + reasons in <30s. Have a fallback recorded video in case of network/laptop issues.
